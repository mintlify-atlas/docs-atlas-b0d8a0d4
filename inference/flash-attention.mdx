Standard attention requires materializing a full N×N matrix of attention scores in GPU memory — a cost that grows quadratically with sequence length and quickly becomes the dominant bottleneck in both training and inference. Flash Attention sidesteps this entirely by never writing the full score matrix to memory at all. Instead, it tiles the computation into small blocks that fit in fast on-chip SRAM, uses an online softmax algorithm to accumulate the output incrementally, and writes only the final result back to slow off-chip memory. The result is attention that is faster, uses far less memory, and scales to sequence lengths that were previously impractical.

## A quick recap of standard attention

The standard scaled dot-product attention formula is:

```
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) × V
```

In practice, computing this for a sequence of length N requires:
1. Computing the score matrix S = QKᵀ — shape (N, N)
2. Scaling S by 1/√dₖ
3. Applying softmax row-wise to get the attention weight matrix A — shape (N, N)
4. Multiplying A by V to get the output — shape (N, dₖ)

Steps 1–3 require storing the full N×N matrix. For N = 2,048 and a typical head dimension, that is over 4 million entries **per attention head**, and they must all live in GPU memory simultaneously.

## Why standard attention is slow

The bottleneck is not floating-point compute — modern GPUs have enormous compute throughput. The bottleneck is **memory bandwidth**: how fast data can be moved between GPU memory subsystems.

GPUs have two levels of memory that matter here:

| Memory | Size | Bandwidth | Access speed |
|---|---|---|---|
| SRAM (on-chip, per SM) | ~20 MB total | ~19 TB/s | Extremely fast |
| HBM (off-chip, global) | 40–80 GB | ~2–3 TB/s | ~10× slower |

Standard attention is **HBM-bound**: every intermediate result — the score matrix, the softmax output — must be written to HBM and read back for the next operation. For a sequence of length N, this produces O(N²) HBM reads and writes. With N in the thousands and many attention layers, this memory traffic dominates total runtime.

The computation itself (the matrix multiplications) is fast. The model spends most of its attention time waiting for data to move between HBM and the compute units.

## The core idea behind Flash Attention

Flash Attention's key insight: if you can compute attention without ever materializing the full N×N score matrix in HBM, you eliminate the dominant source of memory traffic.

The approach:
- Divide Q, K, and V into tiles small enough to fit in SRAM
- Process one tile at a time, accumulating the output in registers
- Use an online softmax algorithm to compute correct softmax without needing the full row
- Write only the final output back to HBM — once

This is called a **fused kernel**: what was previously four separate GPU operations (score computation, scaling, softmax, weighted sum) becomes one CUDA kernel that keeps all intermediate values in SRAM throughout.

<Note>
  Flash Attention is **IO-complexity optimal**: it provably achieves the minimum number of HBM reads and writes required to compute exact attention. Any algorithm that computes exact attention must read Q, K, V from HBM and write the output back — Flash Attention does nothing more than this minimum.
</Note>

## Tiling: breaking the work into small blocks

The tiling strategy processes the attention computation in small blocks of rows (from Q) and columns (from K and V). Each block fits entirely in SRAM, so no intermediate results need to be written to HBM during processing.

<Steps>
  <Step title="Load a tile of K and V into SRAM">
    Select a block of consecutive key and value vectors (e.g., tokens 64–127). Load both into SRAM. These stay resident while all query blocks are processed against them.
  </Step>
  <Step title="Load a tile of Q and compute scores">
    Select a block of query vectors. Compute the dot products between this Q block and the K block already in SRAM. The resulting score tile never leaves SRAM.
  </Step>
  <Step title="Update the online softmax running statistics">
    Use the online softmax trick to update running max and normalization values for each query row. This allows computing the correct softmax incrementally without needing the full row to be available.
  </Step>
  <Step title="Accumulate the output">
    Multiply the (locally normalized) score tile by the V block and add it to the output accumulator for each query row, rescaling the previous accumulator to account for the updated running max.
  </Step>
  <Step title="Move to the next tile and repeat">
    Advance to the next K/V block. Repeat steps 2–4 for all Q blocks. After all tiles are processed, divide each output row by its final normalization sum and write the result to HBM.
  </Step>
</Steps>

## Online softmax: computing softmax without the full row

Standard softmax over a row requires two passes: first to find the maximum (for numerical stability), then to compute exponentials and normalize. This makes it incompatible with tiling, where the full row is never available at once.

Online softmax solves this with a single-pass algorithm that maintains running statistics as new tiles arrive:
- **Running max** (`m`): updated whenever a new block contains a larger value
- **Running sum** (`l`): the denominator, rescaled whenever the running max changes

When the running max is updated, all previously accumulated contributions are rescaled by `exp(old_max - new_max)` to remain consistent. This ensures the final result is identical to standard softmax even though the full row was never in memory.

## Implementation

```python
# Conceptual illustration of Flash Attention tiling
# Real Flash Attention is implemented as CUDA kernels

def flash_attention_conceptual(Q, K, V, block_size=64):
    """
    Illustrates the tiling approach of Flash Attention.
    In practice, this runs as a fused CUDA kernel on SRAM.
    
    Q, K, V: (seq_len, d_k)
    """
    import numpy as np
    seq_len, d_k = Q.shape
    scale = 1.0 / np.sqrt(d_k)
    
    # Output accumulator and softmax normalizers
    output = np.zeros_like(Q)
    row_max = np.full(seq_len, -np.inf)   # running max for numerical stability
    row_sum = np.zeros(seq_len)            # running denominator for softmax
    
    # Tile over key/value blocks (outer loop — kept in SRAM)
    for j_start in range(0, seq_len, block_size):
        j_end = min(j_start + block_size, seq_len)
        K_block = K[j_start:j_end]  # (block_size, d_k) — loaded into SRAM
        V_block = V[j_start:j_end]
        
        # Tile over query blocks (inner loop)
        for i_start in range(0, seq_len, block_size):
            i_end = min(i_start + block_size, seq_len)
            Q_block = Q[i_start:i_end]  # loaded into SRAM
            
            # Compute scores for this tile (never leaves SRAM!)
            scores = Q_block @ K_block.T * scale  # (block_q, block_k)
            
            # Online softmax update
            block_max = scores.max(axis=-1)
            new_max = np.maximum(row_max[i_start:i_end], block_max)
            
            # Rescale previous accumulator
            scale_prev = np.exp(row_max[i_start:i_end] - new_max)
            
            # Compute this block's contribution
            exp_scores = np.exp(scores - new_max[:, np.newaxis])
            block_sum = exp_scores.sum(axis=-1)
            
            # Update running statistics
            output[i_start:i_end] = (
                output[i_start:i_end] * scale_prev[:, np.newaxis] +
                exp_scores @ V_block
            )
            row_sum[i_start:i_end] = (
                row_sum[i_start:i_end] * scale_prev + block_sum
            )
            row_max[i_start:i_end] = new_max
    
    # Normalize output
    output = output / row_sum[:, np.newaxis]
    return output
```

## Recomputation in the backward pass

Training requires gradients, which means the backward pass needs the attention weight matrix A — the same N×N matrix that Flash Attention avoids storing. Rather than materializing and storing A during the forward pass (expensive), Flash Attention **recomputes** it during the backward pass from the stored Q, K, V, and the output statistics (row max and row sum).

Recomputation trades a small amount of extra compute for a large reduction in memory. Because SRAM is fast and the computation is cheap relative to the memory bandwidth saved, this is a favorable trade. The backward pass is also implemented as a fused CUDA kernel with the same tiling approach.

## Flash Attention versions

<Tabs>
  <Tab title="Flash Attention 1 (2022)">
    The original Flash Attention paper by Dao et al. demonstrated that IO-aware attention — tiling to minimize HBM reads/writes — could be faster than standard attention in practice, not just in theory. It reduced attention memory from O(N²) to O(N) and enabled training on sequences up to 4× longer than was previously feasible on the same hardware.

    Key characteristics:
    - Fused CUDA kernel for forward and backward passes
    - Online softmax for correct incremental computation
    - Recomputation in the backward pass to avoid storing the attention matrix
    - ~2–4× faster than standard PyTorch attention for typical sequence lengths
  </Tab>
  <Tab title="Flash Attention 2 (2023)">
    Flash Attention 2 achieved roughly **2× speedup over Flash Attention 1** by eliminating non-matrix-multiply operations that were wasting GPU throughput. On modern GPUs, matrix multiplications can run at peak hardware utilization while other operations (additions, comparisons, rescaling) run much slower. Flash Attention 1 spent a meaningful fraction of time on these non-GEMM operations; Flash Attention 2 reorganized the algorithm to minimize them.

    Key improvements over v1:
    - Reduced the number of non-matrix-multiply FLOPs per output element
    - Reorganized the inner loop order to improve cache locality
    - Added parallelism across the sequence length dimension (not just batch and heads), enabling better GPU utilization for long sequences or small batches
    - Became the default attention implementation in most major training frameworks
  </Tab>
  <Tab title="Flash Attention 3 (2024)">
    Flash Attention 3 targets NVIDIA H100 GPUs specifically, exploiting hardware features that were not available on earlier architectures: Tensor Memory Accelerator (TMA) for asynchronous data movement, and WGMMA instructions for warp-group-level matrix multiplication.

    Key improvements over v2:
    - Overlaps attention score computation and softmax rescaling in a pipelined fashion, hiding the latency of each
    - Uses TMA to prefetch the next tile while computing on the current one
    - Achieves ~1.5–2× speedup over Flash Attention 2 on H100
    - Reaches 75% of H100 theoretical peak FP16 throughput for attention — close to the hardware ceiling

    Flash Attention 3 demonstrates that even for a well-optimized kernel, new hardware generations with new instruction sets can unlock further meaningful gains.
  </Tab>
</Tabs>

## Advantages and impact

Flash Attention's impact on the field has been substantial:

**Longer context.** Quadratic memory complexity was the hard ceiling on trainable sequence length. Flash Attention's O(N) memory removes this ceiling. Models have scaled from 512-token contexts to 8K, 32K, 128K, and beyond — all enabled or accelerated by Flash Attention.

**Faster training.** Attention was a major bottleneck in Transformer training. Flash Attention reduces wall-clock training time for attention by 2–8× depending on sequence length and hardware, directly reducing the cost of large training runs.

**Ubiquitous adoption.** Flash Attention is now used in virtually every major LLM training framework — PyTorch, JAX, Hugging Face Transformers, Megatron-LM, and others — as well as in inference engines like vLLM and TensorRT-LLM. It is not an optional optimization; it is the standard implementation.

**Exact attention.** Unlike approximate attention methods (e.g., sparse or linear attention), Flash Attention computes **exact** standard attention. There is no accuracy trade-off — only speed and memory improvements.
